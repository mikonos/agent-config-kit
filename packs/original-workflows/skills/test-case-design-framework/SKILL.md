# 测试用例设计操作系统

> 蒸馏自：Cem Kaner 的风险/场景/bug taxonomy 路线，Glenford Myers 的经典测试设计原则，Boris Beizer 的覆盖与可测性路线。
> 调研截止：2026-04-23

## 使用说明

**擅长**：
- 设计 Web 前台、移动端网页、后台接口、业务流程类系统的测试用例
- 把用例从“功能步骤清单”提升为“风险探测器”
- 统一使用风险驱动、经典黑盒方法和结构化覆盖三种视角
- 发现 happy path 之外的边界、组合、状态、错误路径与历史缺陷模式
- 解释为什么这批用例值得写、漏了什么、下一步该补什么

**不擅长**：
- 替代具体测试工具语法、接口平台操作教程或管理系统字段说明
- 直接回答 2026-04-23 之后的新框架、新浏览器怪异行为或新平台能力
- 生成“看起来很多、实际上没有判断”的流水账用例

## 框架概览

### 三位作者分别负责什么

| 视角 | 代表人物 | 回答的问题 |
|---|---|---|
| **风险与场景** | Cem Kaner | 哪些用例最值得先写、最可能发现重要问题？ |
| **经典设计方法** | Glenford Myers | 一个合格的基础用例该怎么设计，才不只是证明系统没坏？ |
| **结构覆盖与可测性** | Boris Beizer | 我是否遗漏了关键状态、关键组合、关键路径和关键可测性前提？ |

### 一句话总纲

好的测试用例，不是写得多，而是同时满足三件事：

1. **值得写**
2. **会设计**
3. **没漏大洞**

## 核心工作流

### Step 1: 先做 Kaner 式风险排序

先不要写步骤，先回答：

- 哪个功能最值钱、最脆弱、最常变？
- 哪种错误最贵？
- 哪类用户路径最关键？
- 历史上这类系统最容易出现什么 bug？
- 真实场景里会发生什么误用、极端输入、异常依赖？

输出：
- 高价值场景列表
- 高风险缺陷模式列表
- 优先级排序

### Step 2: 再做 Myers 式基础设计

对每个高价值场景，至少检查：

- 等价类划分
- 边界值分析
- 错误猜测
- 正常流 / 异常流
- 输入 / 输出 / 状态变化

目标不是把 textbook 名词写上去，而是逼自己别只写一条 happy path。

### Step 3: 最后做 Beizer 式结构补洞

检查是否存在：

- 状态机遗漏
- 组合矩阵遗漏
- 路径覆盖黑洞
- 环境/依赖切换遗漏
- 可测性不足导致根本测不透

如果结构上有洞，就不是多加几条 case 能补回来的，可能要先改观察手段、数据控制能力或系统设计。

## 六个核心心智模型

### 模型一：测试用例首先服务于风险，而不是服务于表格

**一句话**：若一个用例不能明显降低关键风险的不确定性，它的价值就值得怀疑。

**来源主线**：Kaner

**应用**：
- 先列风险，再写 case
- 优先写关键业务、关键损失、关键脆弱点相关用例

### 模型二：测试的目标是找错，不是证明没错

**一句话**：一个过于温柔的用例，通常不是好用例。

**来源主线**：Myers

**应用**：
- 每条用例都问一句：它最可能在哪个点把系统打穿？
- 如果答案是“不知道”，说明设计还不够锐利

### 模型三：happy path 只是起点，不是主体

**一句话**：只验证主流程通了，几乎永远不足以代表测过了。

**来源主线**：Myers + Kaner

**应用**：
- 异常输入
- 断点值
- 顺序颠倒
- 中断恢复
- 权限变化
- 数据污染

### 模型四：历史 bug 和缺陷模式是用例设计燃料

**一句话**：最好的用例来源之一，不是想象力，而是缺陷史。

**来源主线**：Kaner

**应用**：
- 建立 bug taxonomy
- 让历史缺陷反哺新用例
- 对同类产品常见错误做预判

### 模型五：覆盖不是数量问题，而是结构问题

**一句话**：50 条 case 也可能只是在同一个角落里重复打转。

**来源主线**：Beizer

**应用**：
- 看状态、条件、路径、组合，而不是只看条数
- 对复杂接口或规则系统做结构化拆分

### 模型六：设计 for testability 和设计用例同样重要

**一句话**：如果系统不可观察、不可控制、不可隔离，再聪明的用例也会被卡死。

**来源主线**：Beizer

**应用**：
- 先确认日志、状态、数据注入、依赖替身、环境切换能力
- 不把“测不出来”误解成“没有问题”

## 九条决策启发式

1. 先写风险地图，再写测试用例。
2. 每个重要场景至少补齐正常流、异常流和边界流。
3. 如果一条用例只能证明系统会工作，通常还不够好。
4. 输入类问题先看等价类，数值和范围问题先看边界值。
5. 复杂流程不要只看步骤，要看状态转换和前置数据关系。
6. 用历史 bug、线上事故、客户投诉反向生成新用例。
7. 当 case 越写越多时，先问是不是结构没建模好，而不是继续堆条数。
8. 对接口测试，重点不是 URL 数量，而是契约、异常、权限、时序和数据一致性。
9. 若系统可测性太差，先补观察和控制手段，再谈高质量用例覆盖。

## 常见场景下怎么用

### PC 端 / 手机端网页

优先顺序：

1. 核心用户路径
2. 跨断点布局与交互差异
3. 输入边界与浏览器行为差异
4. 弱网、刷新、返回、并发点击、中断恢复
5. 与后台接口状态不一致时的前端表现

对应方法：
- Kaner：先找高风险真实场景
- Myers：补齐边界、异常、错误猜测
- Beizer：补齐设备/浏览器/状态组合覆盖

### 后台接口

优先顺序：

1. 契约正确性
2. 权限与身份
3. 参数边界
4. 错误码与异常恢复
5. 幂等性、并发、时序、数据一致性

对应方法：
- Kaner：先看最贵的接口风险和失败代价
- Myers：系统化设计参数与响应断言
- Beizer：检查状态迁移、组合矩阵和依赖路径是否漏测

## 输出格式

当用这个 framework 回答问题时，默认按这个顺序输出：

1. 这个测试对象最关键的风险是什么
2. 应优先覆盖哪些高价值场景
3. 用哪些经典设计方法补足基础 case
4. 结构上还可能漏掉哪些状态/组合/路径
5. 还缺哪些可测性支持

## 流派对比

### Kaner 偏什么

- 风险
- 场景
- bug 模式
- 现实世界缺陷发现

### Myers 偏什么

- 基本功
- 黑盒设计技法
- 反直觉测试目标

### Beizer 偏什么

- 覆盖
- 模型
- 可测性
- 系统性补洞

## 边界与误用

- 不要把这个框架误用成“每条 case 都要写成论文”
- 不要只拿 Myers 的名词清单，忽略 Kaner 的风险排序
- 不要只做 Kaner 的经验判断，忽略 Beizer 的结构黑洞
- 不要把 Beizer 的系统化理解成必须追求穷举

## 示例

**问题**：登录接口要怎么设计测试用例？

**框架回答**：

1. 先做 Kaner 风险排序：账户安全、错误锁定、验证码、设备切换、第三方依赖、异常恢复，哪个代价最高？
2. 再做 Myers 基础设计：用户名/密码等价类，长度边界，空值，特殊字符，错误次数边界，异常返回。
3. 再做 Beizer 补洞：登录前后状态切换、token 生命周期、并发登录、跨端会话、权限缓存、依赖服务波动。

如果这三层没分开，你很容易写出 40 条登录 case，却仍然漏掉真正贵的故障。
