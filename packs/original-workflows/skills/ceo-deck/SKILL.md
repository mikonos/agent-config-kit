# ceo-deck

把材料做成“CEO 听得懂、愿意转发、能当场被说服”的全屏演讲 deck。  
核心不是炫技，而是用真实工作流证据证明一个管理判断。

也可把技术教学稿改成 CEO talk：不逐层讲知识点，而是把概念变成一个现场判断。

## 触发时机

使用本 skill，当用户要：

- 仿照 `https://noa.stevewang.ai/lobster-ceo-talk` 做 deck
- 做 CEO / 创始人 / 高管 / 投资人听的 HTML 演讲稿
- 把 AI 落地、产品案例、客户案例、战略判断讲成现场分享
- 需要“案例 + 截图 + 时间线 + 启示”的可信叙事
- 把技术解释型 HTML deck 改成暗场全屏 CEO talk

如果用户只是要一般 HTML 幻灯片排版，优先用 `html-deck`；如果明确是 CEO 级案例叙事，用本 skill。

## 核心判断

CEO deck 不先讲功能，先讲“为什么这件事改变了管理/组织/产品判断”。  
有案例时，先讲“真实发生的事”；没有案例时，先讲“一个关键误解”，再把概念改写成现场判断。

## 自动识别材料类型

不要让用户选择模式。先读输入材料，自动判断它主要是哪一种：

- **证据型材料**：有人物、客户、截图、时间线、交付物、反馈。
- **概念型材料**：主要是技术原理、方法论、框架、课程讲解、抽象定义。

判断后仍然只产出一种东西：**暗场 CEO talk**。

### 如果是证据型材料

用真实发生的事推进。

每个案例都要回答：谁遇到什么高价值问题，AI/产品如何介入，多久产生什么结果，为什么这改变了组织或决策方式。

### 如果是概念型材料

把课程型、技术型、原理型 deck 改造成 CEO talk。

转换规则：

1. 不按教材目录逐章讲，先找到一个“高管误解”
2. 把技术概念改写成现场判断，例如“System Layer 不是组件，而是控制面”
3. 压缩底层细节，把细节放到 1 页“压缩版”
4. 放大可行动层：用户/管理者真正能设计、购买、治理、复用的部分
5. 用 quote / part / story / contrast / timeline / closing 重新编排，而不是继续堆 pipeline 和 proof cards

概念型典型主线：

1. Cover：主题变成一个工作现场，而不是课程标题
2. Quote：先打破误解
3. Regular：列出新的判断框架
4. Part：章节转场，重新集中注意力
5. Compressed explainer：把必要概念压成 1 页
6. Story：用讲述方式解释主战场
7. Cards：只保留 3 个控制面或决策面
8. Contrast：短 prompt / 旧方式 vs 强上下文 / 新方式
9. Timeline：从 Prompt 到 Skill，或从临时动作到可复用系统
10. Closing：一句能被复述的管理判断

## 叙事骨架

默认结构：

1. 封面：品牌信号 + 一句话主题 + 场景承诺
2. 开场：拒绝理论，承诺展示真实案例
3. 定义：先把核心对象讲清楚，排除误解
4. 高管关切：说明为什么这不是 IT/工具问题，而是管理问题
5. 目录：3-5 个真实案例
6. 案例循环：背景冲突 -> AI/产品介入 -> 过程证据 -> 成果反应 -> 本质判断
7. 规律提炼：把所有案例收束成一个共同机制
8. 行动建议：给听众 3 步起点
9. 误区提醒：阻断常见逃避理由
10. 收尾金句：一句值得带走的话

## 案例页公式

每个案例优先按这个证据链组织：

- 真实角色：人名 / 职位 / 公司 / 场景
- 真实压力：时间、客户、增长、对外表达、组织协同
- 真实输入：截图、文件、表格、采访提纲、群聊消息
- 真实过程：时间线、步骤、AI/产品具体做了什么
- 真实产物：链接、页面、提案、稿件、报告、截图墙
- 真实反应：客户/高管/团队原话
- 企业启示：这件事说明新的工作方式是什么

没有证据时，不要伪装成案例；改成“设想/假设/演示”。

## 真实复盘型 deck：引用与产物（硬约束）

当 deck 要引用真实人物的话、或展示真实产物（聊天/创作复盘、客户案例、过程复盘）：

- **逐字引用必须回源提取，禁止凭记忆或压缩摘要重构。** 要放某人原话，就去读真实来源（对话 transcript 的 `.jsonl`、原始文档、聊天记录）一字导出。记忆和摘要会把「别人/AI 的产出」误记成「他的原话」、把命令格式（如 `/plan …`）的消息整条漏掉。本 skill 一次复盘 deck 就因凭摘要重建，三处「原话」失真，最后回 transcript 才校正。
- **提取真实人类消息的实操**：解析 `.jsonl` 取 `role:user` 文本块；排除 tool_result、压缩摘要、skill 注入；命令消息（`<command-name>`）要解出 `<command-args>` 里的真实文本，否则整条丢失。
- **展示产物，不要描述产物。** 产出版位放真实片段（代码 / 清单 / 回报样例 / 截图），让读者「看得见」，而不是只读到「它做了什么」一句概述。

## 排版系统

默认采用 `lobster-ceo-talk` 型全屏暗色演讲风：

- 画布：100vw x 100vh，全屏翻页，不做长滚动页面
- 字体：标题用中文 serif，正文用 sans；标题要有演讲压场感
- 颜色：暗底，红色做主强调，金色做案例/证据标签；避免五彩散乱
- 层级：小标签 -> 大标题 -> 内容模块 -> 高亮结论
- 对齐：普通内容页左对齐；封面、章节页、金句页居中
- 信息量：每页只做一个主动作，不把定义、证据、结论塞在同一屏
- 导航：底部翻页控件 + 顶部进度条 + 页码
- 不要固定 16:9 居中画布 + 外阴影；这会变回“嵌在网页里的 PPT”
- 不要把图形组件当主角；CEO talk 的主角是判断、节奏和证据

## 页面原语

- Cover：emoji/标识 + eyebrow + H1 + subtitle + author/date
- Quote：居中大引语，用红色强调一个关键词
- Regular：左上小标签，大标题，正文或列表，底部高亮结论
- Part：整页章节转场，红底或强色块，CASE 编号 + 案例标题
- Story：案例背景页，左对齐叙事段落 + 底部 reveal
- Cards：三到四个步骤卡，最后一张可 accent
- Timeline：真实过程记录，时间点 + 动作 + 结果
- Screenshot wall：多张真实截图网格，制造证据压力
- Contrast：左右对比，传统方式 vs 新工作流
- Closing：一句可被复述的最终判断

## 从技术 deck 改版的硬动作

当输入是技术课件或解释型 deck：

1. 去掉 PPT 画布感：`.deck { width: 100vw; height: 100vh; }`，`.slide { position:absolute; inset:0; box-shadow:none; }`
2. 收窄色彩：暗底 + 红 + 金为主；蓝绿只在必要技术图里少量出现
3. 降低图解密度：pipeline、节点图、proof card 不要连续出现；最多保留 1-2 页
4. 把“层级教学”改成“判断推进”：每页标题写结论，不写章节名
5. 增加剧场节奏：每 3-4 页插入 quote 或 part 作为情绪重置
6. 把最后一页改成可复述句，不要只是 takeaway 列表

参考已跑通样式：

- 全屏暗场：`100vw / 100vh`
- 顶部细红进度条 + 右上小页码 + 底部胶囊导航
- Cover 用一个强记忆标识，不用复杂装饰图
- 高密度页用三卡横排或左右 contrast，不用大面积流程图
- 技术解释压缩成“必要背景”，主角交给“工作现场 / 控制面 / 可验收链路”

## 写作原则

- 标题写判断，不写栏目名
- 每页正文少解释，多给证据和动作
- 关键词用红色强调，但每屏只强调少数词
- 案例页多用具体时间：2 分钟、10 分钟、28 分钟、当晚
- 高管启示要落到组织方式、决策链、成本结构或增长速度
- 不说“AI 很强”，说“哪个工作基本单位变了”
- 技术概念要翻译成管理语言：现场、控制面、验收、复用、成本、速度、组织链路
- 没有真实案例时，不要伪造证据；用 contrast 和 timeline 讲清“旧方式 -> 新方式”的改变
- 版位先定职能、再填内容：每个版位（标题/引用/卡格/产出块）先问“这个位置存在是为了干什么”，再决定写什么、放哪段引用——不是先有一段话再找地方塞
- 术语前先给大白话：任何专有名（harness、hard landscape、act-then-surface…）先用一句人话说清它干什么，再可选挂洋名；抽象页（方法论/框架）必须戳回读者前面已见过的具体实例，不空降名词
- 叙事人称统一：复盘型 deck 固定“我＝主角/作者，你＝读者”；别让“你”一会儿指主角、一会儿指读者（最隐蔽的混淆源）

## 交付与隐私

- **单文件可传播**：deck 应“一个文件发出去就能看”。图片一律 base64 内嵌（`data:image/png;base64,…`）；大图先 `sips -Z 420` 压到合理尺寸再编码，避免 HTML 过大；不留外部图片依赖。
- **隐私边界**：含逐字原话、个人微信/二维码、内部路径或项目代号的 deck 属**私有**——只私下分享，**不进公开仓库**。要公开先脱敏（路径换占位符、去人名与内部代号）。

## 验收

- [ ] 第一屏能立刻看出这是给 CEO/高管的演讲，不是普通网页
- [ ] 每页只有一个主判断
- [ ] 每个案例都有真实证据或明确标注为演示
- [ ] 至少有一页时间线或截图墙承担可信度
- [ ] 如果输入主要是概念型材料，至少有一页 contrast 和一页 timeline 承担说服推进
- [ ] 不再像固定 16:9 PPT 嵌在网页里
- [ ] 技术图解没有连续压住 3 页以上
- [ ] 章节转场能让现场听众重新集中注意力
- [ ] 最后一页是一句可以复述的管理判断
- [ ] 手机和桌面视口都不溢出、不遮挡、不需要滚动
- [ ] 引用真人原话的，已逐字回源核对（非凭记忆/摘要）；产出展示的是真实片段而非描述
- [ ] 可一个文件传播（图片已 base64 内嵌）；含隐私/逐字原话的 deck 未进公开仓库

## 与 html-deck 的关系

`ceo-deck` 负责叙事定位、页面节奏、证据结构和 CEO 级表达。  
`html-deck` 负责通用 HTML deck 的视觉 polish、密度控制和风格继承。  
做 CEO 级 HTML deck 时，可以先用本 skill 定骨架，再用 `html-deck` 做 polish。
