# training-deck-refiner

把基础技术讲稿，迭代成“非技术听众能听懂、deck 上能记住、讲稿里能讲清”的培训型 deck 内容。

默认专家组合：
- Karpathy：机制是否准确，术语是否误导
- 费曼：是否能用一句话和一个例子讲懂
- Nancy Duarte：deck 是否只有一个主判断，是否能被现场复述
- CEO deck 视角：高管/投资人是否能快速抓住“这对我有什么用”

## 适用场景

- 技术培训、AI/LLM 培训、投资人/高管听众的解释型 deck
- 用户给出一段讲稿草稿，要求优化讲稿和 HTML deck
- 每页/每层需要从“原理解释”变成“可讲、可记、可复述”的内容
- 用户要求比较讲稿与 deck 是否匹配

不用于普通研究报告、官网、营销 landing page；视觉 polish 优先配合 `html-deck`，CEO 演讲风格配合 `ceo-deck`。

## 核心原则

1. **先机制，后类比**  
   先讲原理骨架，再用类比加深记忆。不要用类比替代机制。

2. **讲稿解释，deck 召回**  
   讲稿放完整机制、例子、边界、术语解释。Deck 只放一个主判断、少量结构和记忆钩子。

3. **每页一个可复述判断**  
   标题要像一句能被听众带走的话，不写栏目名。

4. **术语保留但要落地**  
   可以保留 `token`、`base model`、`SFT` 等必要术语，但必须配一句人话解释。

5. **类比要少而稳**  
   一个模块保留 2-3 个主类比即可。不要每页换类比，避免记忆钩子互相打架。

## 标准工作流

### 1. 明确本页任务

读取当前讲稿片段和 deck 页面，先判断：
- 本页听众要理解的核心机制是什么？
- 听完之后，听众应该能复述哪一句？
- 了解这个机制，对听众有什么用？

### 2. 讲稿优化

讲稿按这个顺序组织：

1. 先给一句定义
2. 解释机制
3. 给一个小例子
4. 说明边界和误区
5. 再用类比压缩记忆
6. 收束到“这对我有什么用”

### 3. Deck 压缩

Deck 页面优先使用一个主标题 + 3-4 个最小信息块。

对非技术听众，优先选下面任一结构：

- 是什么 / 怎么做 / 产物是什么 / 对我有什么用
- 读什么 / 怎么学 / 学成什么 / 对我有什么用
- 输入 / 机制 / 输出 / 对我有什么用
- 误区 / 正解 / 例子 / 带走句

其中“对我有什么用”要落到用户行动：怎么提问、怎么判断输出、怎么设计上下文、怎么避免误用。

### 4. 专家合议检查

输出前按四个问题自查：

- Karpathy：技术说法是否准确？有没有层级错误或过度类比？
- 费曼：一个聪明但非 AI 专业的人能不能听懂？
- Duarte：deck 上是不是只有一个主动作？有没有把口播内容塞上屏？
- CEO deck：这页有没有告诉听众“这改变我怎么用 AI”？

### 5. 输出格式

默认输出：

```md
## 判断
一句话说明当前讲稿/deck 是否成立。

## 讲稿建议
可直接替换的讲稿段落，或需要保留/删除的点。

## Deck 建议
- 标题：
- 副标题：
- 三/四个信息块：
- 底部带走句：

## 风险
可能误导的术语、类比边界、听众会卡住的地方。
```

如果用户要求“直接改”，则直接编辑目标讲稿和 HTML deck，并在改后验证页面没有明显溢出。

## 本轮沉淀的示例模式

### Tokenizer

Deck 主句：
> Token 不只是计费单位，更是 LLM 的文字

讲稿负责解释：token 是模型最小阅读单位，不总等于一个汉字；所有上下文第一步都会变成 token。

### Pre-training

Deck 主句：
> Pre-training 就像熟读唐诗三百首

Deck 信息块：
- 读什么：海量 token，像熟读大量诗文
- 怎么学：反复猜下一个 token，背错了就被纠正
- 学成什么：Base model，像一个只会吟诗、不断续写的脑子
- 对我有什么用：它像知道很多，但不是在查一个资料库

讲稿负责解释 next-token prediction、参数、Transformer/attention、base model 边界。
