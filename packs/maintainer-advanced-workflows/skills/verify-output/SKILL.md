# verify-output（对抗性验证）

> **设计来源**：独立验证者模式；公开版适配 Codex、Cursor 和 Claude Code，
> 不假设某个 Runtime 一定提供特定名称的 Task tool。
> **核心问题**：同一个 agent 既做又审，容易自洽而忽略盲点。独立的"第二双眼睛"能显著提高产出质量。

## 适用场景

- deep-reading 完成后的最终审查
- PRD / 方案产出后的对抗性检验
- 重大决策前的多视角压力测试
- 用户说"帮我验证一下""审查""找找问题""second opinion"

## 工作流

### Step 1：确定审查对象

用户指定要审查的产出（文件路径或对话中的内容）。

### Step 2：选择独立审查面

优先使用当前 Runtime 已提供的独立审查者、subagent 或独立任务，并明确：

- 只读，不修改文件或外部系统；
- 只读取用户指定的审查对象和必要上下文；
- 角色是找出影响正确性或用户要求的问题，不为凑数制造意见。

没有独立审查能力时，优先让用户在新任务或 Fork 中运行同一审查提示。仍无法
分离上下文时可以自查，但必须标注：`审查独立性：自查，非独立审查`。

### Step 3：三种审计模式（可组合）

| 模式 | 视角 | 输出 |
|---|---|---|
| **假设审计** | 列出产出中的隐含假设，标注哪些未验证 | 假设清单 + 验证状态（已验证/未验证/不可验证） |
| **覆盖度审计** | 对照用户原始请求，检查是否全部回答 | 覆盖矩阵（请求点 × 是否被回答） |
| **逻辑审计** | 检查论证链是否有断裂、矛盾、跳跃 | 逻辑链标注（声称→证据→推理，标记断裂点） |

### Step 4：输出 VERDICT

```markdown
## VERDICT: [PASS / ISSUES_FOUND / FAIL]

### 假设审计
- [已验证] ...
- [未验证] ...

### 覆盖度审计
- [已覆盖] 用户请求 A
- [未覆盖] 用户请求 B — 建议补充：...

### 逻辑审计
- [完整] 论证链 1
- [断裂] 论证链 2 — 在 X 到 Y 之间缺少 Z

### 建议行动
1. ...
2. ...
```

## 独立审查请求模板

将以下内容交给当前 Runtime 的独立只读审查面：

```
你是一个独立的验证审查员。你不是产出的作者，你的目标是找到问题而非确认正确。

## 审查对象
[填入文件路径，让子 agent 自行 Read]

## 用户原始请求
[填入用户的原始需求原文]

## 审计任务
1. **假设审计**：读完产出后，列出所有隐含假设，标注（已验证/未验证/不可验证）
2. **覆盖度审计**：逐条对照上方「用户原始请求」，检查每个要求是否被回答
3. **逻辑审计**：检查论证链（声称→证据→推理），标记断裂点

## 输出格式
VERDICT: PASS / ISSUES_FOUND / FAIL
然后逐项列出各审计结果 + 建议行动。
```

## 新任务或 Fork 替代方案

当前 Runtime 没有独立审查者时：
1. 从当前对话 Fork 出新对话
2. 在新对话中粘贴上述审查 prompt
3. 只提供审查所需材料；若 Fork 会继承上下文，仍提醒审查者不要沿用作者结论

## 与 workflow-audit 的区别

| | verify-output | workflow-audit |
|---|---|---|
| **审什么** | 产出的**内容质量**（假设对不对、逻辑通不通、覆盖全不全） | 流程的**执行完成度**（每个 Phase 是否做了、DoD 是否通过） |
| **谁审** | 独立只读子 agent（第二双眼睛） | 默认独立只读子 agent 审流程；主 agent 负责补执行 |
| **何时用** | 产出完成后、决策前、需要"第二意见"时 | deep-reading Phase 6.5 等流程节点 |
| **互补关系** | workflow-audit 确认"做没做"，verify-output 确认"做对没" |

## 触发建议

- deep-reading Phase 5 / Phase 6.5 之后，可选调用本 skill 做独立验证
- 方案/决策产出后，由用户判断是否需要"第二意见"
- 不建议自动触发（成本较高），保持手动按需
